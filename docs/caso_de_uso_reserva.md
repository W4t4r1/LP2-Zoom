# Especificación de Caso de Uso: CU-01 (LP2-Zoom)

Este documento detalla la especificación del caso de uso principal del proyecto **LP2-Zoom** adaptado estrictamente al formato y campos requeridos en la evaluación de la asignatura.

---

## Ficha del Caso de Uso: CU-01

| Campo | Detalle |
| :--- | :--- |
| **Código** | **CU-01** |
| **Nombre** | **Gestión de Sala de Espera, Admisión e Inicio de Videoconferencia con Transmisión en Tiempo Real** |
| **Actor principal** | **Docente / Anfitrión (Host)** y **Invitado / Estudiante (Guest)** |
| **Actores secundarios** | **Servidor de Sockets** (orquestador de red concurrente), **Base de Datos (Supabase Cloud PostgreSQL)**. |
| **Objetivo** | Permitir al Anfitrión moderar la cola de espera de invitados en tiempo real y, tras admitirlos, iniciar oficialmente la videoconferencia para habilitar la transmisión interactiva de video y chat. |
| **Precondición** | 1. El Anfitrión y el Invitado deben haber iniciado sesión en sus respectivos clientes (sus credenciales deben estar validadas con hash SHA-256 en Supabase).<br>2. La conexión TCP a través del puerto 5000 entre los clientes y el servidor debe estar activa y estable (Singleton `ClienteConexion`). |
| **Postcondición** | 1. La solicitud del Invitado en la tabla `SolicitudesSala` de Supabase pasa a estado `"ACCEPTED"`.<br>2. El Invitado se registra en la tabla `ParticipantesSala` como participante activo.<br>3. Ambos clientes transicionan a la pantalla `REUNION` mediante el gestor gráfico `CardLayout`.<br>4. Se inicia de forma concurrente la captura, transmisión y decodificación de video (`CAMERA_FRAME` en Base64) y el chat. |
| **Flujo principal** | **1.** El Anfitrión (Docente) crea una sala de videoconferencia y el sistema le genera un código alfanumérico único de 6 caracteres.<br>**2.** El Invitado (Estudiante) ingresa el código de 6 caracteres en su selector de rol y hace clic en "Unirse".<br>**3.** El cliente del Invitado envía una trama JSON de tipo `JOIN_ROOM_REQUEST` con el código y su ID de usuario a través del socket TCP.<br>**4.** El Servidor recibe la trama, registra la solicitud en Supabase con estado `"PENDIENTE"` y notifica al Anfitrión con una trama `WAITING_ROOM_UPDATE`.<br>**5.** El Anfitrión visualiza al Invitado en su lista de espera en pantalla (actualización segura en el EDT de Swing mediante `SwingUtilities.invokeLater`) y hace clic en "Admitir".<br>**6.** El cliente del Anfitrión envía una trama JSON `ADMIT_USER` con estado `"ACCEPTED"`. El servidor actualiza el registro en la tabla `SolicitudesSala` e inserta al participante en `ParticipantesSala` como activo.<br>**7.** El Servidor envía al Invitado la trama `ADMIT_USER` con estado `"ACCEPTED"`, y este transiciona a la pantalla de espera admitido.<br>**8.** El Anfitrión presiona "Iniciar reunión", enviando la trama `START_MEETING_REQUEST`. El servidor difunde la trama `MEETING_STARTED` a todos los clientes.<br>**9.** Ambos clientes conmutan sus pantallas a la tarjeta `REUNION` e inician asíncronamente la captura y decodificación de video (`CAMERA_FRAME`) y la mensajería en red. |
| **Flujo alternativo** | **5a. El Anfitrión rechaza la solicitud:**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. En el paso 5, el Anfitrión hace clic en "Rechazar".<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El cliente envía `ADMIT_USER` con estado `"REJECTED"`. El servidor actualiza Supabase a `"REJECTED"` y notifica al Invitado.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El cliente del Invitado sale de la pantalla de espera y regresa al selector principal con el mensaje "Solicitud rechazada".<br><br>**6a. El Invitado cancela la solicitud de espera:**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El Invitado hace clic en "Cancelar" mientras está en la pantalla de espera.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El cliente envía `LEAVE_ROOM` y el servidor elimina la solicitud en Supabase. |
| **Reglas de negocio** | **RN-01 (Regla de Oro de Persistencia):** Las credenciales de base de datos no se guardan en el cliente; todas las validaciones de acceso se ejecutan a través de tramas JSON validadas en el servidor.<br>**RN-02 (Hilo Gráfico Protegido):** Toda actualización de red debe empujarse al EDT mediante `SwingUtilities.invokeLater()` para evitar bloqueos de la UI.<br>**RN-03 (Código Único):** La sala debe identificarse por un código alfanumérico único de 6 dígitos generado asíncronamente en el servidor. |
| **Datos de entrada** | Tramas JSON del protocolo:<br>- `JOIN_ROOM_REQUEST` (roomCode, userId)<br>- `ADMIT_USER` (roomCode, userId, estado)<br>- `START_MEETING_REQUEST` (roomCode) |
| **Datos de salida** | Tramas JSON del protocolo:<br>- `WAITING_ROOM_UPDATE` (lista de solicitudes pendientes)<br>- `MEETING_STARTED` (orden de inicio de videoconferencia)<br>- `CAMERA_FRAME` (frames de cámara en Base64)<br>- `CHAT_MESSAGE` (mensajes del historial) |
| **Excepciones** | **EX-01: Pérdida de conexión (Socket cerrado):**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. Si se pierde la conexión TCP con algún cliente en cualquier etapa, el servidor lanza una `IOException`, libera el socket, actualiza el estado de participación en Supabase a `"SALIÓ"` y notifica al resto para apagar los feeds de video.<br><br>**EX-02: Cámara física bloqueada:**<br>&nbsp;&nbsp;&nbsp;&nbsp;1. Si la webcam del cliente falla al iniciarse, `CameraProxy` conmuta a la estrategia simulada para evitar una caída del programa. |
